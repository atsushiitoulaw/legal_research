# コーパス投入スクリプト（ほうりつ探検隊）

個人情報保護法の条文を、条・項単位に分解して Azure に投入するスクリプトです。

## 構成

| ファイル | 役割 |
|---|---|
| `parse_law_xml.py` | e-Gov法令XML を条・項単位のチャンクに分解し、JSONL を出力 |
| `upload_to_azure.py` | JSONL を Blob Storage に保管し、ベクトル化して AI Search に登録 |

## 出力されるチャンクの形

要件定義書 3-2 のメタデータ設計に対応しています。

```json
{
  "id": "law-27-5",
  "source_document": "個人情報の保護に関する法律",
  "article": "第27条第5項",
  "section": "",
  "caption": "第三者提供の制限",
  "content": "（第三者提供の制限）\n次に掲げる場合において、当該個人データの提供を受ける者は、前各項の規定の適用については、第三者に該当しないものとする。\n一　個人情報取扱事業者が利用目的の達成に必要な範囲内において個人データの取扱いの全部又は一部を委託することに伴って当該個人データが提供される場合\n..."
}
```

**設計のポイント**
- **1チャンク＝1項**（号は本文に含める）。要件定義書のとおり、チャンクが条をまたがないようにしています
- `article` が出典表示用のラベル（例：`第27条第5項`）になります
- 法令には節番号がないため `section` は空。ガイドラインでは、ここに `3-4-4` のような節番号が入ります

---

## ガイドライン（通則編）への応用

通則編は PDF なので、XML のように構造が機械的に取れません。ただし、**同じ JSONL 形式にできれば、`upload_to_azure.py` はそのまま使えます**。

つまり、通則編でやることは：

1. PDF から、節（`3-4-4` など）ごとにテキストを切り出す
2. 以下の形の JSONL を作る（`upload_to_azure.py` が読める形）

```json
{
  "id": "gl-3-4-4",
  "source_document": "個人情報保護法ガイドライン（通則編）",
  "article": "",
  "section": "3-4-4 委託先の監督",
  "content": "（該当節の本文テキスト）"
}
```

3. `upload_to_azure.py` に、その JSONL を渡すだけ

```bash
python upload_to_azure.py tsusokuhen.jsonl
```

**注意点**
- `id` は法令と重複しないように、`gl-` のような接頭辞をつけてください
- 1チャンク＝1節を目安に。節をまたぐと、出典が特定できなくなります
- 節の切り出しは AI に手伝わせることもできますが、**区切りが正しいか（3-4-4 の範囲が本当に合っているか）は、必ず人の目で確認してください**
